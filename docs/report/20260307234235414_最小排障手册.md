# CodexManager 最小排障手册

生成时间：2026-03-07 23:42:35
适用范围：桌面端、service、web、协议兼容、发布后现场问题定位

## 1. 先看什么

先确认三件事：

1. 当前请求到底打到的是 `/v1/chat/completions` 还是 `/v1/responses`
2. 当前是桌面端内嵌 service，还是独立 `codexmanager-service` / `codexmanager-web`
3. 是否已经拿到可复现的请求日志、trace 日志、probe 输出

推荐最小自检顺序：

1. 跑统一回归入口：`pwsh -NoLogo -NoProfile -File scripts/tests/gateway_regression_suite.ps1 -Base http://localhost:48760 -ApiKey <key> -Model gpt-5.3-codex`
2. 若只想测工具调用：`pwsh -NoLogo -NoProfile -File scripts/tests/chat_tools_hit_probe.ps1 -Base http://localhost:48760 -ApiKey <key> -Model gpt-5.3-codex`
3. 若要同时看 chat / responses 流式：`pwsh -NoLogo -NoProfile -File scripts/tests/codex_stream_probe.ps1 -Base http://localhost:48760 -ApiKey <key> -Model gpt-5.3-codex`

## 2. `/v1/responses` 不通

优先看：

1. 上游是否真的支持 `responses`
2. 当前客户端是否其实走了 `chat/completions` 转发
3. 请求路径是否被错误改写

建议定位：

1. 先用 `codex_stream_probe.ps1` 看 `responses_headers.txt` 和 `responses_stream.txt`
2. 再看网关 trace 日志里是否出现：
   - 入站路径是 `/v1/responses`
   - 上游路径是否仍然是 `/v1/responses`
   - 是否进入了聚合/映射分支
3. 若 Cherry 可用但手工直调不可用，优先排查客户端是否实际请求了另一条接口

## 3. `chat tools` 不命中

优先看：

1. 请求体里是否真的带了 `tools`
2. `tool_choice` 是否强制指定了目标工具
3. 返回里是否丢了 `tool_calls`

建议定位：

1. 先跑：`chat_tools_hit_probe.ps1`
2. 看输出里的：
   - `tool_hit`
   - `finish_reason`
   - `tool_name_returned`
   - `restored_name_ok`
3. 若非流式命中、流式不命中，优先检查 SSE -> JSON 聚合和 tool name restore
4. 若两边都不命中，优先检查上游模型是否当前支持 tools

## 4. `upstream challenge blocked`

这通常不是本地转发代码直接崩了，而是上游被 Cloudflare/WAF 或鉴权策略拦住。

先查：

1. 账号登录态是否已过期
2. 是否需要 `CODEXMANAGER_UPSTREAM_COOKIE`
3. 是否启用了请求头收敛模式
4. 是否同一账号在多个代理间漂移

建议处理：

1. 刷新账号认证
2. 在设置页或环境变量里补 `CODEXMANAGER_UPSTREAM_PROXY_URL`
3. 必要时补 `CODEXMANAGER_UPSTREAM_COOKIE`
4. 检查 `CODEXMANAGER_CPA_NO_COOKIE_HEADER_MODE`

## 5. `upstream timeout`

先区分是哪种超时：

1. connect timeout
2. total timeout
3. stream timeout

重点检查：

1. `CODEXMANAGER_UPSTREAM_CONNECT_TIMEOUT_SECS`
2. `CODEXMANAGER_UPSTREAM_TOTAL_TIMEOUT_MS`
3. `CODEXMANAGER_UPSTREAM_STREAM_TIMEOUT_MS`
4. 当前代理是否可达

如果只在流式请求超时：

1. 先跑 `codex_stream_probe.ps1`
2. 看 `chat_stream.txt` / `responses_stream.txt` 是否有前半段内容
3. 若有前半段再断，多半是流式阶段超时或上游中途断流

## 6. `codexmanager-web` 启动 `101` / Tokio panic

若直接运行 `cargo run -p codexmanager-web` 或 `codexmanager-web.exe` 后立即退出，并出现类似：

`Cannot drop a runtime in a context where blocking is not allowed`

说明：

1. 启动前同步初始化被放进了 async runtime
2. `reqwest::blocking` 相关逻辑在 runtime 内 drop

当前主线已修复该问题。若再出现：

1. 先确认运行的是最新构建产物
2. 再确认启动前的同步设置加载没有被重新塞回 async `main`

## 7. 桌面端卡在“正在启动服务”

优先检查前端控制台和启动链：

1. 浏览器模式打开 `http://127.0.0.1:5173/`
2. 看控制台是否有初始化阶段 JS 异常
3. 看桌面端是否真的拿到了 service ready

历史上出现过的典型原因：

1. `isTauriRuntime` 被拆丢，导致启动阶段直接抛错
2. 刷新任务链里有同步异常，遮罩没被收起

建议处理：

1. 跑前端测试：`pnpm -C apps run test`
2. 跑桌面端 lib 测试：`cargo test --manifest-path apps/src-tauri/Cargo.toml --lib`
3. 再执行 `cargo tauri dev`

## 8. 发布前最小检查

发布前至少跑：

1. `pwsh -NoLogo -NoProfile -File scripts/tests/rebuild.test.ps1`
2. `pwsh -NoLogo -NoProfile -File scripts/tests/assert-release-version.test.ps1`
3. `pwsh -NoLogo -NoProfile -File scripts/tests/gateway_regression_suite.test.ps1`

若改了协议适配：

1. 再跑一次真实联通的 `gateway_regression_suite.ps1`

## 9. 现场回收什么文件

统一回归入口默认会产出：

1. `summary.json`
2. `summary.txt`
3. `chat_tools_non_stream.txt`
4. `chat_tools_stream.txt`
5. `codex_stream_probe/*`

有问题时优先让用户回传整个输出目录，而不是只截一张图。
